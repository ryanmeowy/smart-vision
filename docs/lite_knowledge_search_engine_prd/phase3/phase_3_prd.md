# PRD 3：Engineering Enhanced（收敛版）

## 1. 目标定义

Phase 3 不再扩功能面，聚焦三件事：

* 可重建
* 可调优
* 可观测

目标是把系统从“可用”推进到“可运维、可复盘、可持续演进”。

## 2. 范围与优先级

### 2.1 P0：局部重建能力（必须做）

要求：

* 支持按 `assetId` 重建
* 支持按类型重建（仅文本 / 仅图片）
* 支持失败后续跑（断点续跑或失败项重试）
* 保留任务状态、进度、失败原因

价值：

* 避免全量重建
* 降低运维成本
* 提升线上可修复性

### 2.2 P0：检索参数实验化（必须做）

要求：

* 参数配置化并可按实验组切换：
  * `recallTopK`
  * `rerankWindowTopN`
  * `fusionAlpha/fusionBeta`
  * OCR 开关
* 固定评测集，输出可复盘对比结果

价值：

* 具备真实“搜索调优”能力
* 让策略选择有数据依据

### 2.3 P0：指标面补齐（必须做）

要求：

* 指标从“有埋点”升级为“可分析问题”，至少覆盖：
  * 搜索总耗时 P50/P95
  * recall、rrf、rerank 分段耗时
  * rerank 调用量/成功率/降级率
  * 重建任务成功率、重试率、平均耗时
  * 命中来源占比（vector/text/ocr/tag）

价值：

* 能定位瓶颈和退化来源
* 支撑发布前后回归判断

### 2.4 P1：版本化策略（按当前架构收敛）

当前系统已按 embedding profile 区分索引（provider/model/dimension/preprocess）。
因此本阶段版本化优先级定义为：

* 必做：
  * 维持“索引版本 + alias 切换”作为主版本边界
  * ingest job 记录 `pipelineVersion`（审计/回溯）
* 可选：
  * `segment.version` 字段仅在“同一索引混跑多版本”场景下引入

说明：

* 若继续采用“一版本一索引”，`segment.version` 不是必需项。

## 3. 实验与报告要求

至少完成 2 组正式对比，建议：

* `hybrid` vs `hybrid + rerank`
* OCR `off` vs OCR `on`

每组输出：

* Recall@K / MRR(NDCG 可选)
* P50/P95 延迟
* 成本或调用量变化
* 结论与上线建议

## 4. 非目标（本阶段不做）

* 视频能力
* 多租户 / SaaS 权限体系
* 在线学习排序
* 大规模前端平台化改造

## 5. 验收标准（Phase 3 Done）

满足以下条件视为完成：

1. 支持按 `assetId` / 类型的局部重建，并可追踪任务状态
2. 检索参数可配置并可按实验组运行
3. 指标可用于定位性能与效果退化
4. 至少两组实验有量化结论与调优记录
5. 版本边界策略明确（索引版本主导，job 侧可追溯）

## 6. 产出物

* 局部重建接口与任务流说明
* 参数化配置清单（含默认值与回退策略）
* 指标清单与观测面板（或统计报表）
* 实验报告（含结论与取舍）
* 版本化策略文档（索引/alias 与 pipelineVersion）
