## 一、拆分原则

* **不重做**现有的图片入库主链路
* **不重做**现有任务状态、重试、RRF、rerank、explain、metrics
* **只新增**文本入库、统一 segment、text + image 检索闭环

## 2.3 建议执行顺序

| 顺序 | 建议先做          | 原因                         |
|---:|---------------|----------------------------|
|  1 | P1-01 ~ P1-04 | 先把文本资产接进现有任务体系             |
|  2 | P1-05 ~ P1-09 | 先生成文本 chunk，不然后面都没法测       |
|  3 | P1-10 ~ P1-13 | 统一 segment 是 Phase 1 的真正底盘 |
|  4 | P1-14 ~ P1-16 | 文本语义召回要在统一索引后接入            |
|  5 | P1-17 ~ P1-20 | 搜索 API 收口                  |
|  6 | P1-21 ~ P1-23 | 前端和验收最后做，避免 UI 牵着后端跑       |

# 四、取舍

这阶段最容易失控的地方有两个：

### 1. PDF 解析

别一开始追求“完美页内阅读顺序恢复”。
Phase 1 先做到：

* 页级提取
* 基础 chunk
* 能返回页码
  就够了。

### 2. 统一模型

别一开始就搞 asset-level / segment-level / nested-level 三套检索模型。
Phase 1 直接统一到 `KbSegmentDocument`，最省时间。
