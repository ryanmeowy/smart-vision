# 2. 入库侧模型关系

```text
                ┌──────────────────────┐
                │      IngestJob       │
                │──────────────────────│
                │ jobId                │
                │ assetId              │
                │ jobType              │
                │ status               │
                │ currentStep          │
                │ errorMessage         │
                │ createdAt            │
                └─────────┬────────────┘
                          │ 1:N
                          │
                          ▼
                ┌──────────────────────┐
                │      IngestStep      │
                │──────────────────────│
                │ stepId               │
                │ jobId                │
                │ stepName             │
                │ status               │
                │ payload              │
                │ errorMessage         │
                │ startedAt            │
                │ finishedAt           │
                └──────────────────────┘
```

### 说明

不用重做整套任务体系，只是把**文本入库**接进现有的 job/step 体系里。

对 Phase 1 来说，文本任务最小步骤可以是：

```text
UPLOAD
  ↓
PARSE_TEXT
  ↓
SPLIT_CHUNK
  ↓
GENERATE_EMBEDDING
  ↓
INDEX_SEGMENT
  ↓
SUCCESS / FAILED
```

# 3. Segment 类型图

```text
SegmentType
├── TEXT_CHUNK
├── IMAGE_CAPTION
└── IMAGE_OCR_BLOCK
```

### 含义

* `TEXT_CHUNK`：文本切块后的检索单元
* `IMAGE_CAPTION`：图片整体描述
* `IMAGE_OCR_BLOCK`：图片里的 OCR 文本块

这三个已经足够支撑 Phase 1。

# 5. 实体职责

## Asset

原始资源中心，负责：

* 文件身份
* 存储位置
* 资源类型
* 生命周期状态

## Segment

检索中心，负责：

* 被搜索
* 被排序
* 被解释
* 被引用

## IngestJob / IngestStep

流水线中心，负责：

* 状态管理
* 重试
* 可观测性

## SearchResultHit

展示中心，负责：

* 给前端返回统一结果
* 聚合 snippet / pageNo / explain / preview
