# 多轮对话检索专项（二期）任务卡

依据：`多轮对话检索_二期PRD.md`  
更新时间：2026-05-02  
状态：Draft for Execution

## 1. 阶段目标

把现有“对话检索 API + Streamlit 验证页”升级为可用的对话式知识检索产品闭环：

1. 用户可在类似 ChatGPT 的界面中连续对话。
2. 每轮返回 LLM 摘要总结和 Top3 asset 结果卡片，每张卡片包含 `primaryHit` segment。
3. 点击卡片可进入预览页，定位命中位置并继续浏览全文。
4. 保留传统关键词检索入口。
5. 全链路具备基础观测与验收脚本。

## 2. 边界原则

### 2.1 Must

1. 后端响应扩展 `resultCards`，每个卡片对应一个 asset，且包含 `primaryHit` segment。
2. 摘要继续复用 `answer` 字段，保证证据约束和引用可回溯。
3. 提供 `GET /api/v1/preview/segments/{segmentId}` 预览接口。
4. Streamlit MVP 只做输入、answer、resultCards 基础展示。
5. React/Next.js 正式版完成对话页、关键词检索入口、预览页基础能力。
6. 覆盖观测事件与核心验收用例。
7. 提供后端会话历史列表 API，左侧历史不依赖浏览器本地缓存，但只承诺 TTL 内恢复。
8. Top3 卡片必须是 asset 级展示结果，不能丢失 `primaryHit.segmentId`。
9. 预览接口必须通过 `segmentId` 稳定解析资源，不依赖入库期短 TTL 缓存。
10. React/Next.js 必须完成统一 API Client、鉴权、baseURL 与错误态处理。
11. Top3 选择必须按 asset 聚合，不让同一 asset 默认占满 Top3。
12. 预览响应支持 `surroundingChunks`，用于 TXT/MD 上下文展示与定位兜底。
13. 图片 bbox 缺失或无效时必须安全降级，不绘制错误框；bbox 精准框选作为后续版本必做增强。
14. `resultCards` 必须复用当轮 `retrievalResult.topCandidates` 生成并持久化，禁止二次检索。

### 2.2 Out of Scope

1. 不做语音输入和语音播报。
2. 不做复杂文档标注编辑。
3. 不做 Agent 工具调用编排。
4. 不做复杂权限体系重构。
5. 不要求本期实现流式打字机效果。
6. 不做长期记忆、用户偏好画像、跨会话自动记忆。
7. 不在 Streamlit 内投入产品级复杂预览体验，Streamlit 只承担 MVP 基础展示。
8. 不为会话删除/重命名单独引入 DB；长期会话持久化另开专项。
9. 不做首次进入对话检索页的新手引导。
10. 不做 Playwright 自动化 E2E。
11. 不做上一处/下一处命中导航。
12. 不做合规级日志脱敏专项。

## 7. Phase 2 Done 标准

满足以下条件视为二期完成：

1. `POST /api/conversations/{sessionId}/messages` 返回证据约束摘要和最多 3 个 `resultCards`。
2. Top3 均为 asset 级卡片，每张卡片必须包含 `primaryHit.segmentId`。
3. 同一 asset 的多个命中进入 `additionalHits`，不额外占用 Top3 卡片位。
4. `GET /api/conversations?limit&cursor` 支持左侧会话历史 TTL 内恢复，并按用户隔离。
5. 会话删除/重命名基于 Redis 可落地，不要求二期引入 DB。
6. `GET /api/v1/preview/segments/{segmentId}` 可返回短期预览 URL 或受控内容地址、类型、anchor 与 `surroundingChunks`。
7. 预览接口通过 `segmentId` 查询稳定索引或持久化元数据，不依赖入库期短 TTL 缓存。
8. Streamlit MVP 能演示输入、answer、resultCards 基础展示。
9. React/Next.js 正式版支持对话检索、关键词检索和预览页。
10. React/Next.js 具备统一 API Client、鉴权接入、baseURL 配置与统一错误态。
11. PDF/TXT/MD 至少各有一组可通过的预览定位样例；IMAGE 可打开原图并展示命中文本。
12. bbox 缺失或无效时安全降级，不绘制错误命中框；bbox 精准框选后续版本补齐。
13. 观测事件覆盖发送、卡片点击、预览成功、预览失败。
14. 最小日志安全检查通过：不打印完整签名 URL、token、大段原文。
15. 二期接口文档、交互文档和验收记录完整。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| React/Next.js 工程从零引入，学习和搭建成本不确定 | 正式版延期 | M2 先用 Streamlit 完成可演示闭环，React 骨架并行推进 |
| Top3 与摘要引用不一致 | 用户信任下降 | C2-01 固定选择规则，C2-06 校验引用能映射到 `primaryHit/additionalHits` |
| 同一 asset 多个高分 segment 占满 Top3 | 用户看到的来源单一 | C2-01/C2-03A 按 asset 聚合，同 asset 其他命中进入 `additionalHits` |
| 预览 URL 过期 | 点击失败率升高 | C2-32 自动刷新一次预览 URL |
| TXT/MD 精确定位不稳定 | 定位体验不一致 | MVP 先 snippet 匹配，失败按 chunkOrder 近似定位 |
| PDF text layer 高亮受文件影响 | 只能页级定位 | 页级定位作为硬验收，高亮作为增强体验 |
| bbox 数据缺失或异常 | 图片定位不完整或画错框 | 无 bbox/无效 bbox 时展示原图和命中文本，不做伪定位 |
| 左侧历史依赖本地缓存 | TTL 内刷新、换设备或清缓存后历史丢失 | C2-07A 新增后端会话历史列表 API；超过 TTL 不承诺恢复 |
| 为删除/重命名引入 DB 导致范围膨胀 | 二期延期 | C2-00A 明确二期先用 Redis 用户索引，长期历史另开 DB 持久化专项 |
| 直接加载对象存储 URL 受 CORS/Range Request 限制 | PDF/TXT/MD 预览不可用或不稳定 | C2-01A、C2-09B 先验证加载策略，必要时走后端受控代理/内容 API |
| 预览依赖入库期 Redis metadata | 缓存过期后旧结果无法预览 | C2-09A 通过 `segmentId` 查询稳定索引或持久化元数据 |
| surroundingChunks 响应过大或越权 | 性能和安全风险 | C2-08A 限定 previous/current/next、单 chunk 限长，并复用权限校验 |
| React/Next.js 鉴权和错误处理分散 | 接口联调和线上错误态不一致 | C2-19A 统一 API Client、baseURL、token/header 和错误拦截 |
| 日志记录完整签名 URL、token 或大段原文 | 安全风险 | C2-33 做最小日志安全检查，不做合规级脱敏专项 |
| 历史卡片回放触发二次检索 | 结果漂移、成本增加 | C2-03A/C2-05 要求 `resultCards` 复用当轮候选并随 turn 持久化 |
