## 1. **以 segment 为中心**

* 文本按 chunk 切分
* 图片按 caption / OCR block / region 切分
* 视频按 transcript segment 切分
* 检索单位就是这些细粒度片段

这类架构的核心优点是：

* 召回粒度细
* 容易直接返回“第几页 / 哪一段 / 哪个 OCR block”
* 很适合 RAG 和问答，因为 grounding 单元天然就是片段

```text
用户 Query
   ↓
Query Rewrite / Embedding
   ↓
统一检索层（以 segment 为检索单元）
   ├── TEXT_CHUNK
   ├── IMAGE_CAPTION
   └── IMAGE_OCR_BLOCK
   ↓
RRF / Rerank
   ↓
命中 segment
   ↓
回挂父资源 Asset / Document
   ↓
返回：
- 文本：文件名 + 页码 + 片段
- 图片：图片名 + 命中摘要 + 预览
```

这一路的核心思路是：**先把内容切成小片段，再检索片段**。
Azure 的文档分块和向量化指南，本质就是这条路线：把文档拆成 chunk，chunk 独立处理、独立向量化，再作为检索单元参与搜索。([Microsoft Learn][1])

### 适合场景

* 知识库搜索
* RAG
* 需要返回精确片段
* 需要页码、段落、OCR block 级定位

### 优点

* 定位天然细
* 很适合后续问答和引用
* text + image 比较容易统一到一个检索协议里

### 缺点

* 容易打碎上下文
* 对复杂版面、图表、扫描页，纯 chunk 可能损失布局信息

## 2. **以 document / asset 为中心**

也可以叫：

* **document-centric**
* **asset-centric**
* 在视觉文档里经常还会落成 **page-centric**

它的思路不是先把世界切碎，而是先把“完整资源”当作主检索对象：

* 一本书是一个 document
* 一张图是一个 asset
* 一个 PDF 页是一个 page
* 一个视频是一个 asset

查询时，先召回“哪个文档/哪一页/哪张图”最相关，
然后再在文档内部找局部证据，或者把整页/整图直接交给后续模型处理。

```text
用户 Query
   ↓
Query Rewrite / Embedding
   ↓
统一检索层（以 document / asset / page 为检索单元）
   ├── 整本文档
   ├── 整张图片
   └── 整个 PDF 页面
   ↓
RRF / Rerank
   ↓
命中文档 / 页面 / 图片
   ↓
文档内定位 / 二次抽取
   ↓
返回：
- 哪份文档最相关
- 哪一页最相关
- 哪张图最相关
- 再从内部找片段
```

这一路的核心思路是：**先找对资源，再在资源内部定位**。
这类路线在**复杂 PDF、图表、表格、扫描页、视觉文档检索**里非常常见。
比如 ColPali 这类方法，就是把**文档页面的图像表示**直接拿来做检索，而不是先把页面纯文本化再切 chunk；官方博客明确把它定位成“solely using the image representation of document pages”。([Hugging Face][2])

### 适合场景

* PDF 页面检索
* 扫描件
* 图表、表格、复杂版面
* 视觉文档检索

### 优点

* 保留页面布局和视觉结构
* 对复杂文档常常更稳
* 更适合“先找页，再看页内内容”

### 缺点

* 粒度天然更粗
* 要想返回精确片段，通常还要二次定位
* 对你这种 text + image 统一知识检索，工程上会更绕一点

---

## 一句话解释

### Segment-centric

“**先切碎，再检索碎片**”

### Document/asset-centric

“**先找对文档/页面/资源，再在里面定位**”

---

# 这两类架构各自适合什么

| 维度           | Segment-centric | Document / Asset-centric |
| ------------ | --------------- | ------------------------ |
| 检索粒度         | 细               | 粗到中                      |
| 定位能力         | 强，天然适合页码/片段返回   | 需要二次定位                   |
| 对长文档问答       | 很适合             | 需要额外内部定位                 |
| 对复杂版面/图表/扫描页 | 容易丢结构           | 更有优势                     |
| 对实现复杂度       | 中               | 中到高                      |
| 对你现在的项目      | **更合适主线**       | 更适合做补充层                  |

## 3. **分层/父子架构（hierarchical, parent-child）**

也就是：

* 顶层按 document / page / image asset 建模
* 底层按 chunk / OCR block / region 建模
* 检索时可能先召回 segment，再回挂 parent
* 或者先召回 document/page，再下钻局部

```text
用户 Query
   ↓
Query Rewrite / Embedding
   ↓
第一层召回
   ├── segment 召回
   └── document / page / asset 召回
   ↓
融合 / 重排
   ↓
选中最佳父资源 + 最佳子片段
   ↓
返回：
- 最相关资源
- 资源内最相关片段
- 片段引用与解释
```

这其实就是：
**检索中心用 segment，组织中心用 asset/document/page。**
Azure 的分块文档也强调，要根据文档结构做 semantic chunking 和 layout-aware chunking；这类思路很容易在工程上落成“父资源 + 子片段”的分层模型。([Microsoft Learn][1])

当前项目最合适的是：

```text
主检索中心：segment
父资源中心：asset
展示中心：asset + best segment
```

也就是：

* **检索时**：按 `TEXT_CHUNK / IMAGE_CAPTION / IMAGE_OCR_BLOCK`
* **组织时**：回挂到 `Asset`
* **展示时**：给用户看“最相关资源 + 资源内最佳片段”

这条路最适合现在的目标，因为：
* 做 text + image 统一检索
* 返回页码和片段
* 后面接对话式检索和 grounding
* 还要尽量复用你现有图片检索底盘

所以一句话总结：

**Segment-centric 是你的主架构，Asset-centric 是你的父级组织层。**
不是二选一，而是**以 segment 为检索中心，以 asset 为展示和管理中心**。
