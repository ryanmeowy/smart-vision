# ES CLI 正式化迁移方案（脚本版 -> 正式 CLI）

更新时间：2026-04-10

## 1. 背景

- 当前已具备可用的脚本版 CLI：`scripts/sv-es`。
- 现状优势：验证速度快、命令模型已稳定。
- 现状不足：分发与升级能力弱、可测试性一般、跨平台体验不统一。

## 2. 迁移目标

1. 提供正式 CLI 可执行产物（版本化、可分发、可升级）。
2. 保持与脚本版命令兼容（登录、查询、输出、错误提示）。
3. 降低团队使用门槛（安装统一、行为稳定、文档清晰）。

## 3. 技术路线（建议）

- 语言与框架：Java + Picocli（与现有后端技术栈一致，维护成本低）。
- 打包方式（V1）：`fat jar`（优先快速交付）。
- 可选增强（V2）：GraalVM native image（启动快、分发体验更好）。

## 4. 兼容性策略

1. 命令兼容：保持现有命令结构不变。
- `login/logout/whoami`
- `cluster health/stats`
- `index list/stats`
- `mapping get`
- `doc get/search`

2. 配置兼容：延续 profile 配置。
- 路径：`~/.smartvision/escli/<profile>.conf`

3. 错误兼容：保留当前错误码友好提示语义。

## 5. 功能范围

### V1（本次迁移）

1. 命令与参数解析（对齐脚本版）。
2. 登录态管理（本地 profile 存储与读取）。
3. HTTP 调用封装（统一 header、超时、重试策略）。
4. 输出渲染（`json/table`）。
5. 错误处理（非 200 退出码 + hint + requestId）。
6. 打包与使用文档。

### V2（后续）

1. 自动补全（zsh/bash completion）。
2. 自动升级与版本检查。
3. native image 分发。

## 6. 非功能要求

1. 安全：token 文件权限最小化（600）。
2. 可观测：CLI debug 日志可开关。
3. 稳定性：请求超时、网络错误提示清晰。
4. 性能：单次命令启动与响应时间可接受。

## 7. 风险与对策

1. 风险：命令行为与脚本版不一致。
- 对策：建立“脚本版 vs 正式版”契约回归用例。

2. 风险：打包后跨环境运行差异。
- 对策：优先 jar 分发，后续再做 native 优化。

3. 风险：用户迁移成本。
- 对策：保留脚本版一个过渡周期，并提供迁移说明。

## 8. 里程碑建议（2-4 天）

1. M1（0.5 天）：工程骨架与命令框架。
2. M2（1 天）：核心命令 + 登录态 + HTTP 调用。
3. M3（0.5-1 天）：输出渲染 + 错误映射。
4. M4（0.5-1 天）：测试、打包、文档、试运行。

## 9. 验收标准（DoD）

1. 正式 CLI 覆盖现有 7 个查询命令与 3 个会话命令。
2. 输出与错误行为与脚本版一致（核心场景）。
3. 可通过统一命令运行并完成登录后查询。
4. 提供安装、升级、回滚文档。
5. 完成最小回归测试清单并通过。
