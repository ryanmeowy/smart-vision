你在编写代码时，必须严格遵守以下规范与原则：

一、总体原则
1. 以模块化、组件化为基本设计思想，确保系统结构清晰、边界明确。
2. 保持高内聚、低耦合。一个模块应只负责自己职责范围内的事情，不要让多个模块相互缠绕、相互渗透。
3. 遵循单一职责原则。一个类、一个函数、一个组件只做一件事，并把这件事做好。
4. 代码必须优先追求简洁、高效、可读。不要为了“看起来全面”而牺牲可理解性。
5. 设计应当贴合当前真实需求，避免为了未来不确定的场景提前堆砌复杂抽象。

二、设计要求
1. 不要过度设计。
   - 不要为了小问题引入过多层级、模式、抽象和中间结构。
   - 不要在需求尚不明确时，预先设计复杂扩展点。
   - 能直接解决的问题，就不要绕远路。

2. 优先做恰到好处的设计。
   - 既不要写成一团糟，也不要为了“架构感”而人为复杂化。
   - 抽象必须有明确价值，能减少重复、提升可维护性、明确边界时才进行抽象。
   - 如果某个抽象只是“也许以后会用到”，那通常不应该现在引入。

3. 保持实现路径清晰。
   - 控制调用链长度，避免逻辑跳转过多。
   - 避免无意义的封装、嵌套和转发。
   - 命名、目录结构、模块划分都应服务于“让人快速看懂”。

三、代码实现要求
1. 不要写太多兜底代码。
   - 不要为了追求“绝对不会报错”而加入大量无意义的判空、默认值、降级分支、吞异常逻辑。
   - 错误处理要有依据、有边界、有业务意义。
   - 只处理真实会发生、且值得处理的异常场景。

2. 不要写冗余代码。
   - 相同或高度相似的逻辑，应提炼复用。
   - 但不要为了消除一点点重复，就引入反而更难理解的抽象。
   - 复用应服务于可维护性，而不是机械追求“零重复”。

3. 不要写猜测性代码。
   - 不要预判用户未来可能有十种需求，然后一次性把十种方案都写进去。
   - 不要实现当前没有被明确提出、也没有被现有上下文证明必要的功能。
   - 代码应面向当前需求和可确认的演进方向，而不是面向想象。

4. 保持代码直接、稳定、易改。
   - 优先选择简单、成熟、容易理解的实现方式。
   - 不要炫技，不要为了“高级”而使用不必要的技巧。
   - 后续维护者应该能快速定位、理解并修改代码。

四、可读性要求
1. 命名要准确、清晰、符合语义，避免含糊不清的变量名、函数名、类名。
2. 函数应尽量短小，逻辑层次分明，避免单个函数承担过多流程控制与业务细节。
3. 条件分支、循环、异常处理要尽量简洁，避免嵌套过深。
4. 注释应解释“为什么这样做”，而不是机械复述“代码正在做什么”。
5. 输出的代码风格要统一，结构要整齐，便于审阅和维护。

五、职责与边界要求
1. 严格区分不同层的职责，不要把表示层、业务层、数据层逻辑混写在一起。
2. 组件之间通过清晰的接口协作，不要直接依赖对方内部实现。
3. 工具类、公共方法、基础组件只承载真正通用的能力，不要塞入业务特例。
4. 每个模块都应有明确存在理由，避免“什么都能做一点”的万能模块。

六、沟通与确认要求
1. 对需求、边界、字段含义、业务规则存在不明确之处时，必须先询问。
2. 不得自行脑补需求，不得擅自补全未经确认的业务逻辑。
3. 不得为了“保证代码一定能运行”而编写一堆不必要的兼容逻辑、模拟逻辑、猜测性逻辑。
4. 当信息不足时，优先指出缺失项，并给出需要确认的问题，而不是擅自决定。
5. 如果存在多种实现方案，应结合当前场景选择最合适、最克制的一种，而不是默认最复杂的一种。

七、输出要求
1. 输出代码时，以满足当前需求为目标，避免无关扩展。
2. 除非明确要求，否则不要额外生成大量测试桩、演示代码、兼容层、配置样板。
3. 除非明确必要，否则不要引入新依赖、新框架、新设计模式。
4. 代码应做到：
   - 结构清晰
   - 依赖合理
   - 职责明确
   - 易于阅读
   - 易于维护
   - 易于迭代

八、最终目标
你的首要目标不是“把所有可能性都考虑进去”，而是：
在充分理解需求的前提下，用尽可能简单、清晰、合理的方式，写出满足当前场景的高质量代码。

请始终记住：
- 简洁优于炫技
- 清晰优于复杂
- 合理抽象优于过度设计
- 明确确认优于主观猜测
- 面向当前真实需求优于面向虚构未来
